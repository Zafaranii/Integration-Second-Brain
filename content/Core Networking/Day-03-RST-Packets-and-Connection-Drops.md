# Day 3 — RST Packets and Connection Drops

> [!info] Why This Day Exists
> "Connection reset by peer" is one of the most common error messages in middleware logs — and one of the least understood. RST packets are TCP's emergency brake. Knowing what triggers them and how to find them in a packet capture means you can diagnose these failures in minutes instead of hours.

## Theory

### RST vs FIN — The Two Ways a Connection Ends

|                | FIN                                  | RST                                  |
| -------------- | ------------------------------------ | ------------------------------------ |
| Meaning        | "I'm done sending, graceful close"   | "Abort this connection immediately"  |
| Handshake      | 4-way (FIN/ACK/FIN/ACK)              | Single packet, no acknowledgement    |
| State cleanup  | TIME_WAIT on active closer           | Immediate — no TIME_WAIT             |
| Data in flight | Other side can still send after FIN  | All buffered data discarded          |
| Who sends it   | Application calls close() gracefully | OS or application on error condition |

RST is TCP's emergency abort. No negotiation, no cleanup, no TIME_WAIT. The connection simply ceases to exist the moment the RST is sent.

---

### What Causes a RST

#### Cause 1 — No Listener on the Port

Most immediate and obvious. Client sends SYN → OS on the target machine has nothing listening on that port → OS immediately sends RST.

Error: `Connection refused` (this is a RST, not a timeout — it fails **fast**).

```
Client → SYN → Target OS
Target OS → RST → Client    (no SYN-ACK, immediate RST)
```

#### Cause 2 — Application Crashes or Abruptly Exits

Application was running, had established connections. Application process is killed with SIGKILL (`kill -9`) or crashes. The OS, noticing the socket owner died, sends RST on all its open connections.

This is why you see `Connection reset by peer` in ACE logs when a backend pod is killed in OpenShift during a rolling deployment — the old pod's connections are RST'd as the OS cleans up.

#### Cause 3 — Firewall or Load Balancer Kills an Idle Connection

The most common cause in a banking environment. A firewall or NAT device has a connection tracking table with an idle timeout. When a connection is idle longer than that timeout, the device removes the entry from its table. The next packet on that "dead" connection gets a RST from the firewall (or is simply dropped — see Day 2 idle timeout discussion).

> [!warning] This Is the Root Cause of "Works Fine Then Fails After Idle"
> If you see `Connection reset by peer` that happens reliably after a connection has been idle for N minutes, a firewall idle timeout is almost certainly responsible. See [[Day-02-Connection-Timeouts-vs-Read-Timeouts]] for the fix (set pool idle timeout shorter than firewall's).

#### Cause 4 — SO_LINGER with Timeout 0

An application can explicitly set the `SO_LINGER` socket option with a timeout of 0. When the application then closes the socket, the OS sends RST instead of FIN, skipping the graceful close sequence entirely. Some servers do this intentionally to avoid TIME_WAIT accumulation. Not a problem in itself — just explains why you occasionally see RST even from a "healthy" server.

#### Cause 5 — Out-of-Order or Invalid Segment

If a packet arrives on a connection with a sequence number that the receiver considers invalid — wrong sequence range, invalid flags — the receiver may send RST to abort. This can happen with connection state desynchronisation (rare on normal networks) or NAT issues.

#### Cause 6 — Half-Open Connection Detection

A machine reboots and loses all connection state. The other side still thinks a connection is established and sends data. The rebooted machine receives data for a connection it has no record of → sends RST.

This is why MQ channel reconnection after a queue manager restart produces `Connection reset by peer` on the other end.

### Error Messages Generated by RST

| Error Message              | Context               | What It Means                                          |
| -------------------------- | --------------------- | ------------------------------------------------------ |
| `Connection reset by peer` | ACE, Java, any socket | Remote sent RST after connection was established       |
| `Connection refused`       | curl, ACE, any socket | Remote sent RST in response to SYN (nothing listening) |
| `Broken pipe`              | Writing to a socket   | Tried to send data after the remote already sent RST   |
| `ECONNRESET`               | System call level     | Same as "connection reset" — the errno code            |
| `ECONNREFUSED`             | System call level     | Same as "connection refused"                           |

### RST in Your Stack Specifically

| Scenario                                                 | RST source                                                       | Error in logs                                                         |
| -------------------------------------------------------- | ---------------------------------------------------------------- | --------------------------------------------------------------------- |
| ACE calls backend API; backend pod restarted             | OpenShift killing the old pod → OS RSTs its connections          | `Connection reset by peer` in ACE logs                                |
| MQ channel drops after queue manager restart             | Remote QM lost connection state, RSTs next data packet           | Channel status: `STOPPED`, MQ error MQRC 2009 (connection broken)     |
| DataPower pool connection idle >5 min; firewall kills it | Firewall RSTs the next packet sent on that connection            | DataPower backend error log, or `Connection reset` returned to client |
| Kafka consumer disconnects from broker abruptly          | Broker RSTs the consumer's connection on next heartbeat interval | `Disconnected from node`, subsequent reconnect                        |
| ACE → Oracle JDBC; Oracle session timeout hit            | Oracle server RSTs the JDBC connection                           | `IO Error: Connection reset` in ACE JDBC node                         |

## Hands-on

### Exercise 1 — Capture and Identify RST Packets

```bash
# Start a capture
sudo tcpdump -i lo -n 'tcp' -w /tmp/rst-demo.pcap &

# Trigger a connection refused RST
curl http://localhost:19876 || true   # nothing on this port

# Trigger a RST from SO_LINGER 0 (Python server that abruptly closes)
python3 -c "
import socket, struct
s = socket.socket()
s.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
s.bind(('localhost', 18765))
s.listen(1)
conn, _ = s.accept()
# Set SO_LINGER: l_onoff=1, l_linger=0 → RST on close
import struct
conn.setsockopt(socket.SOL_SOCKET, socket.SO_LINGER, struct.pack('ii', 1, 0))
conn.close()
s.close()
" &

sleep 0.5
curl http://localhost:18765 || true

kill %1 2>/dev/null
wait 2>/dev/null

wireshark /tmp/rst-demo.pcap
```

**In Wireshark:**

- Filter: `tcp.flags.reset == 1`
- For the connection refused RST: no SYN-ACK precedes it — RST comes immediately after SYN
- For the SO_LINGER RST: connection was established (you'll see the handshake), then RST instead of FIN

### Exercise 2 — Simulate Firewall Idle Kill (RST after idle)

```bash
# Start a simple echo server
nc -lk 7777 &

# Connect a client and establish connection
exec 3<>/dev/tcp/localhost/7777

# Verify ESTABLISHED
ss -tn | grep 7777

# Now simulate the firewall killing the connection by directly resetting it
# We'll use iptables to drop all traffic on port 7777 temporarily
sudo iptables -A INPUT -p tcp --dport 7777 -j REJECT --reject-with tcp-reset

# Try to send data on the "dead" connection
echo "hello" >&3
# → Should produce: bash: echo: write error: Broken pipe

# Clean up
sudo iptables -D INPUT -p tcp --dport 7777 -j REJECT --reject-with tcp-reset
kill %1
exec 3>&-
```

### Exercise 3 — Read RST in ACE Logs

Look at your ACE integration server logs for `Connection reset` or `ECONNRESET`. For each occurrence:

1. What flow and node generated it?
2. What backend was it calling?
3. What time of day? Is there a pattern (e.g. always first request after a quiet period)?
4. Does it self-recover on retry?

If it self-recovers on retry and happens after idle periods → firewall idle timeout. If it happens during high load → backend crashing under pressure.

## Key Takeaways

- RST = immediate abort. No cleanup, no TIME_WAIT, no negotiation.
- "Connection refused" = RST in response to SYN. Fast failure. Something is wrong with the target, not the network path.
- "Connection reset by peer" = RST on an established connection. Remote aborted — crash, restart, or firewall kill.
- "Broken pipe" = you tried to write after RST was already received.
- In banking environments, the most common RST cause is **firewall idle timeout** killing pooled connections. Set your pool idle timeout shorter than the firewall's.
